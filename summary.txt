Hex — Iteration 2 summary
=========================

This iteration delivered the requested changes and features. All backend tests pass (35),
both projects typecheck and build, and the full Docker Compose stack was verified end-to-end.

1) Persistence: SQLite -> PostgreSQL + Drizzle ORM
--------------------------------------------------
- Removed bun:sqlite. State now lives in PostgreSQL, accessed via Drizzle ORM (schema and
  queries in TypeScript; no hand-written SQL; migrations generated with drizzle-kit and run
  automatically on startup).
- Why Postgres: an external networked DB makes server restarts/upgrades safe (state isn't
  trapped in a process), gives the best analytical-query story for match history, and is the
  seam for running multiple backend instances later.
- Dependency injection retained and extended: the `Store` interface (Persistence/Store.ts)
  has `DrizzleStore` (prod) and `InMemoryStore` (tests). Tests need no database.
- Schema (Persistence/Schema.ts): Participants, Matches, Moves.
- Active games stay in memory for low latency and are written through to Postgres on every
  change (match row + append-only move rows in one transaction).

Dev database (standalone container, host port 5433):
  docker run -d --name hexpg -e POSTGRES_PASSWORD=hex -e POSTGRES_USER=hex -e POSTGRES_DB=hex \
    -p 5433:5432 postgres:16-alpine
  DATABASE_URL=postgres://hex:hex@localhost:5433/hex   (this is the in-code dev default)

Compose integration (done): docker-compose.yml now has a postgres:16-alpine service with a
named volume (HexData) and a healthcheck; the backend gained
DATABASE_URL=postgres://hex:hex@postgres:5432/hex and depends_on postgres (service_healthy).
Migrations run on backend startup. Moving dev -> compose was configuration only; no code
changes. Verified: `docker compose up --build` brings up postgres (healthy) -> backend
(migrates + boots) -> frontend (nginx proxies /Ws, /Api, /Health).

2) Move history (for analysis)
------------------------------
- Every move and swap is recorded append-only in the Moves table: ordinal, colour, kind
  (Place/Swap), row/col, clock remaining after, think time (ms), timestamp.
- Exposed for analysis at GET /Api/Matches/:id/Moves.

3) Swap (pie) rule
------------------
- Implemented in GameInstance: after the first move the game enters AwaitingSwap; the decider
  may Swap (take over the opening — participants exchange colours, board stays) or decline by
  simply playing a move. New `Swap` client message; GameView carries Phase + CanSwap.

4) Match-creation options
-------------------------
- GameSettings gained Colour preference (Red/Blue/Random) and a SwapRule toggle. Honoured for
  invites and engine challenges; random queue assigns colours randomly. Frontend lobby
  exposes all of it.

5) PlayerId -> Participant model (accounts-ready, no login yet)
--------------------------------------------------------------
- Introduced a Participant hierarchy: HumanParticipant (max 1 concurrent match) and
  EngineParticipant (unlimited) — the inheritance design you suggested. Matches reference
  participant ids; humans are lazily persisted on first sight. This is the slot accounts plug
  into later; no login/account features were added.

6) Engines (foundations + orchestrator)
---------------------------------------
- Managed (in-process) engines: EngineOrchestrator drives a move via an EngineDriver whenever
  it is the engine's turn. EngineSession is the per-game "instance" the orchestrator manages
  (a subprocess/remote driver can slot behind the same interface later). Bundled stub:
  FirstFreeDriver (plays the first empty cell). A default managed engine is auto-provisioned.
- External / self-hosted engines: any player registers one (POST /Api/Engines/External -> a
  one-time token). The engine connects over WebSocket (/Ws?EngineId=..&Token=..), receives
  game-scoped EngineState, and replies with MakeMove/Swap carrying a GameId. Verified: an
  external engine played a full game via token auth.
- Admin-registered managed engines: POST /Api/Engines/Managed with X-Admin-Token (set the
  ADMIN_TOKEN env to enable).
- Note for later (as you flagged): an orchestrator that launches/supervises engine OS
  processes/instances is a natural extension of EngineSession.

7) Clock rehydration across restarts
------------------------------------
- On boot, active games are rehydrated and their clocks are rebased to the load time, so a
  player to move is never charged for server downtime. This is what makes swapping servers
  for an update safe. (With state in Postgres, the whole story is now robust.)

Environment variables
----------------------
Backend:  PORT (3001), DATABASE_URL (dev default postgres://hex:hex@localhost:5433/hex),
          ADMIN_TOKEN ("" disables managed-engine registration).
Frontend: VITE_BACKEND_HTTP / VITE_BACKEND_WS (default: page origin; dev set in
          .env.development to localhost:3001).
Compose:  postgres POSTGRES_USER/PASSWORD/DB=hex; backend DATABASE_URL points at the
          postgres service; frontend published on :8080.

Kept out of scope (unchanged): no accounts/login, no analysis UI beyond the moves endpoint,
no ranking, and no in-process engine OS-process supervisor yet.


Hex — Iteration 3 summary (hardening)
=====================================
All 40 backend tests pass; both projects typecheck/build; the live compose stack was
re-verified after these changes.

1) Random match presets
   Random matchmaking is restricted to 3 server presets (Blitz 11×11 3+2, Rapid 13×13 5+5,
   Classic 11×11 10+10) to stop queue fragmentation. JoinQueue now carries a PresetId; presets
   are exposed via GET /Api/Limits. Invite and engine play keep full options.

2) Ranked-ready matchmaker
   Added a QueueStrategy seam (Matchmaking/QueueStrategy.ts) with RandomQueueStrategy over a
   per-preset waiting pool. Swap in a rating-aware strategy later to make random -> ranked; no
   other changes needed. (Behaviour today is unchanged: random.)

3) Engine play
   Engine games are forced to unlimited time (server overrides the requested clock). Engines
   now declare a supported board-size range (Min/MaxBoardSize on the engine record + a new
   migration); challenges outside the range are rejected. The driver interface carries the
   range; FirstFree supports the full 5–19. The client clamps the board selector to the chosen
   engine's range and hides the clock for engine play. Verified live: out-of-range challenge
   rejected; Fischer request coerced to unlimited.

4) Abandonment cleanup
   A match is forfeit (WinReason "Abandoned") when the player to move is idle past
   ABANDON_TIMEOUT_MS in unlimited games, or a participant stays disconnected that long in any
   game. Implemented as: GameInstance.CheckDeadlines (clock flag OR idle), plus a
   disconnect-forfeit monitor in SessionService. Threshold is a single env var
   (ABANDON_TIMEOUT_MS, default 120000), kept as one knob for a future admin dashboard. Added
   to docker-compose backend env.

Compose: backend now publishes port 3001 to the host (direct access to backend-only endpoints
and the dev frontend / self-hosted engines), in addition to the internal nginx proxy.

Backend
-------

   {Color, Kind, Row, Col}). GameInstance keeps an in-memory MoveLog (appended on place/swap) and

--------

- Ranking/elo is intentionally not implemented, but the seams are in place (per-participant

  valid session) / Login / ListUsers / DeleteUser. Public POST /Api/Register and /Api/Login.
- ParticipantProfile now carries Username + RegisteredAt; MatchSummary carries OpponentId so the
  frontend can link to an opponent's profile.
- Admin: GET state includes Users; DELETE /Admin/Api/Users/:id; admin page has a Users card

Async managed engines — a slow engine can no longer block the server (todos12.md)
- Matches were already driven off the request path (an opponent's move fires OnGameChanged, which only
  schedules the engine on a timer and returns), but the decision itself — EngineDriver.ChooseMove — was
  synchronous, so a heavyweight engine that actually thinks (deep search, a subprocess, or a service
  call) would block Bun's single event loop and stall every other game and WebSocket message while it
  ran. ChooseMove may now return a Promise (its type became Awaitable<EngineChoice | null>); the trivial
  built-in engines stay synchronous and unchanged, since a plain value satisfies Awaitable.
- EngineSession.Act now awaits the choice and, crucially, re-validates the position afterwards: it
  snapshots the ply via the new GameInstance.CurrentMoveCount() before thinking, and after the (possibly
  async) think drops the result unless the game is still active, still the engine's turn, and at the very
  same ply. So a move an engine decides on while the game finishes, is abandoned, or moves on underneath
  it is never applied to a stale board. The orchestrator clears its per-game Pending guard before acting
  (so two managed engines keep chaining moves) and wraps Act in try/catch — a managed engine's think is
  arbitrary user code and must never take the server down or wedge a game.
- Tests: a gated async driver (its think parked on a hand-released Promise) proves the engine is driven
  without blocking and its move lands once released, and that a move it decides on after the human
  resigns mid-think is dropped with the board left untouched (76 pass, 0 fail).
- Process: feature branch feature/async-managed-engines in the Backend submodule, merged to master.

Security review — fixed an unauthenticated live-state IDOR (todos12.md)
- Reviewed every externally reachable entry point (HTTP routes, WebSocket messages, admin routes, the
  DB layer) and the client's input handling, reproducing each suspected issue against the running dev
  build before accepting it. Full write-up in SecurityReport.md.
- High, fixed: GET /Api/State?PlayerId= returned ComputeState for ANY participant id with no token. A
  participant id is not secret (it keys the public profile pages), so anyone could read another player's
  private live state — most damagingly their open invite Code, and since JoinInvite needs only the code,
  hijack their match. Proven live: an attacker knowing only the victim's public id pulled the invite code

  same ownership proof the WebSocket demands) — 400 without a token, 401 on a wrong one, 200 only for the
  owner. The official client never used the endpoint (state arrives over the WS), so nothing legitimate
  breaks. Covered by Tests/HttpAuth.test.ts and re-verified live (attacker 400/401, owner 200).
- Lower-severity items recorded as recommendations, not bugs: no rate limiting on login/register (bcrypt
  cost is the current mitigation); /Api/Matches/:id/Moves is world-readable (UUID game ids, replay is an
  intended public feature); the unbounded admin board-size ceiling is an admin-gated memory-DoS lever;

  parameterisation (no SQLi), bcrypt + hashed tokens, move authorisation bound to the socket identity

Hex — Iteration 20 summary (engine.md): engine analysis pipeline + two real
external engines. engine-analysis.md (authored earlier as a feasibility study)
scoped how an engine could judge a move "good/bad" — a win-probability delta
between the played move and the engine's best — but nothing was wired up. This
iteration implements that whole pipeline plus the engines to drive it. WHY: the
replay viewer was deliberately built leaving room for analysis; the engine
abstraction was the right shape (EngineView position wrapper, async non-blocking
driving, external-socket transport, Drizzle store, one-fetch client replay) and
just needed an analysis capability added, not a rework.

Shared evaluator (Backend/Source/Engines/Evaluation.ts)
- No neural net, no training: the classic shortest-connection-distance heuristic.
  For a colour, the cheapest way to join its two edges costs one move per EMPTY
  cell the path still needs (own stones free, opponent stones are walls); computed
  with a 0-1 BFS. Red joins top<->bottom, Blue left<->right. EvaluatePosition maps
  the distance gap (with a half-move tempo for the side to move) through a logistic
  squash to a win probability. RankMoves does a one-ply search (play each empty
  cell, evaluate from the opponent's seat, flip) and returns the best moves, with a
  faint centre bias only to break ties. AnalyzeMatch reconstructs the board ply by
  ply (mirrors the client's BuildReplayView: a swap exchanges seats, placements
  carry their owner's colour) and emits one PositionAnalysis per position.

Engine contract + managed Heuristic engine
- EngineDriver gained an optional AnalyzeMatch(Moves, BoardSize, SwapRule) ->
  PositionAnalysis[]. Presence alone marks an engine analysis-capable (synced to a
  new SupportsAnalysis flag on the participant at boot). New managed engine
  Heuristic.ts both PLAYS (best shortest-path move; steals a strong opening when a
  swap beats its best reply) and ANALYSES (reuses Evaluation) — so analysis works
  out of the box with nothing self-hosted, and it doubles as a stronger built-in
  opponent than FirstFree/Random.

AnalysisService + persistence + HTTP
- AnalysisService picks a provider (explicit id, else a connected external analysis
  engine, else the managed Heuristic), serves a cache hit, or computes: managed via
  the driver, external by sending AnalyzeRequest over the engine socket and awaiting
  the matching AnalysisResult (RequestId-correlated, 30s timeout). Results persist
  in a new MatchAnalyses table keyed by (GameId, ProviderId) so compute is paid
  once. New protocol messages AnalyzeRequest (server->engine) / AnalysisResult
  (engine->server), parsed and routed in MessageParser/SessionService. Endpoints:
  GET /Api/Matches/:id/Analysis (cache-only, lists providers) and POST (compute +
  persist; 404 unknown/active, 503 no provider). Admin external-engine registration
  (page + API) gained a SupportsAnalysis toggle. Drizzle migration 0005 adds the
  table and the Participants.SupportsAnalysis column.

Frontend overlay (MatchReplay.tsx, HexBoard.tsx) — lazy, one extra fetch
- An "Analyse" button (shown when a provider exists) GETs any cached analysis up
  front and POSTs to compute on demand. Three new, purely client-side renderings of

  (?!/?/?? from the win-probability the mover gave up), and the engine's suggested

Hex — Iteration 25 summary (NeuralEngine: value-loss overfit + replay staleness). After two long
training runs (256-ch net, 11x11) the pattern was clear from the logs: policy loss falls
monotonically and the arena keeps promoting (the engine strengthens), but value loss (vloss) bottoms
out around gen 3-5 then creeps upward for the rest of the run. run1 (shared trunk, value_loss_weight
1.0) hit vloss 0.155 by gen 14; run2 (separate policy/value trunks, w=3.0) reached the same only by
~gen 29 and produced the stronger engine — so the separate-trunk fix worked, but the creep persists,
just slower.

WHY it happens (two mechanisms, same direction). (1) The reported vloss is TRAINING loss on buffer
samples, and the target distribution is non-stationary: as self-play sharpens, games become decisive
and value targets spread across the full ±1 range, so the irreducible MSE floor rises — part of the
"degradation" is the value problem getting harder, not the head getting worse (the tell: ploss keeps
falling and the arena keeps promoting). (2) Real overfitting: each game contributes ~45 positions that
all share ONE outcome label, so the count of INDEPENDENT value labels per gen is ~768 (games), not
~45k (positions); a big buffer churned ~4x/gen lets the value head memorise those shared labels.

Four levers added, all config knobs (env > hyperparams.toml > default), all defaulting to the OLD
behaviour so nothing changes unless opted in:
- Held-out val_vloss instrument (train.py _fresh_value_loss): each gen, before training, measure the
  net's value MSE on the FRESH self-play samples it has NOT trained on yet, logged next to vloss on
  the [train]/[summary] lines. train≈val => rising floor (benign); train≪val => memorisation. This is
  the instrument the other levers are tuned against — you can finally see which mechanism dominates.
- VALUE_TARGET_BLEND (selfplay): the value label z = blend·(±1 outcome) + (1−blend)·(MCTS root value)
  was hardcoded 0.5/0.5; now a knob. Lowering toward the lower-variance, MCTS-averaged root value cuts
  target variance — lower floor, less memorisation. Bumping MCTS 400->600 makes root value a better
  target, so the two compound.
- Value-head dropout (NET_VALUE_DROPOUT, model.py): dropout on the value-head hidden layer, inactive
  at inference (self-play/engine unaffected). Directly fights outcome memorisation. A behaviour/arch
  change (fresh start), which is why it lands with run3.
- value_loss_weight: the 3.0 was a crutch for shared-trunk value starvation; separate trunks have no
  gradient competition to overcome, so the run3 preset drops it to 1.5.

Replay buffer staleness (replay_buffer.py — the other issue found in the logs). A fixed 500k buffer
(~11 gens at steady-state ~45k samples/gen) sampled uniformly means the newest, best generation is
only ~9% of training and gen-1 random-net data lingers ~11 gens; enlarging 200k->500k for diversity
had HALVED the adaptation rate (200k was ~22% newest, ~4 gens). Fix WITHOUT losing the diversity that
also regularises vloss: the buffer is now organised into per-generation blocks with recency-weighted
sampling (REPLAY_RECENCY_HALFLIFE) — a sample's draw weight halves every N generations of age, so
recent positions dominate while old ones stay RESIDENT for diversity instead of waiting to evict.
Whole-generation eviction under the RAM cap; legacy flat snapshots still load; halflife 0 = the old
uniform behaviour.

run3 (cuda-5090 preset) bakes the pack: value_dropout 0.3, value_target_blend 0.35, value_loss_weight
1.5, replay_recency_halflife 3.0 (mcts 600 already in). Verified on the 2070: full smoke_test (incl.
the real 2-gen CUDA training loop), test_logging and test_inference_server all green; a new CPU unit
test covers recency weighting / uniform / whole-gen eviction / legacy snapshot load; a real run shows
val_vloss logging next to vloss (0.24 vs 0.20 — the small, healthy fresh/train gap of an un-overfit
net). Knobs default to old behaviour, so existing presets/runs are unaffected.
